由transform不为none产生的stacking context
  • 2014-10-13
  • CSS

起因

今天同事碰到了一个奇怪的问题，使用了position: fixed的元素居然不以viewport为 containing block，而是以它的父元素为 containing block。原因是因为在设置了fixed的元素的父元素上，用了transform: translate3d();

后来Google了一番，看到 MDN 上列出的几个会生成 stacking context 的原因，其中就有它。我记得以前在前端观察看过这篇关于z-index的那些事儿，其中列出了几个会形成 stacking context 的条件：

  • 当一个元素是文档的根元素时（元素）
  • 当一个元素有一个position值而不是static，有一个z-index值而不是auto
  • 当一个元素有一个opacity值小于1

当初就是看了这篇文章，才对 stacking context 有了初步了解。不过现在看来，它还少一个条件：

  • 当一个元素有一个transform属性，且值不为none时

为何有这个困惑

今天看browserify的文档，看到exports的两种写法，不禁起了困惑。

其文档是这么说的，当只需要导出单独的一样东西，可以这么写：

    
        module.exports = function(){}; // 可以是一个函数或者其他东西。

但是，这样写绝对不可以：

    
        exports = function(){}; // 可以是一个函数或者其他东西。

快速获取图片宽高
  • 2014-09-25
  • JavaScript

开发过一个等宽瀑布流，请求后端的数据发现后端并没有提供图片宽高，应该是后端的哥们忙吧。加之项目比较赶，还是想办法在前端解决吧。

那时的解决方法就是直接监听请求来的每一组的数据的图片的onload事件，触发一次或者img.complete === true就记一次数，发现加载完了，就把图片按照瀑布流的布局显示出来。因为无法预知每张图片到底高度多少，所以不知道瀑布流接下来该怎么摆，所以整个过程是阻塞的，也就是图片不加载完，就不干其他事。
