无限滚动更新的痛

发表于 2015-02-09 | 分类于 用户体验

  1. 今天突然有人提出，对于一个无限下拉更新的图片浏览网站来说，如果点开了某张图片，跳转到这张图片的详情页面。再回退回来，想找回刚才那张图片，发现又回到页面顶部，又得重新开始拉啊拉啊拉，几乎等于这玩意儿没办法用。

  2. 搜了几个类似类型的网站，第一个是有演出，他们大概也遇到了同样的问题。解决方法是，干脆就不用下拉更新了，还原到「上一页，下一页」的模式。应该只要记住当前数据加载的页码，就能很容易找回数据。而且，回退后，自动滚动到上一次看到的图片（item） 也是可以做的。

  3. 第二个是Pinterest，也是这种类型网站的鼻祖了。点开某张图时，首先会弹出一个弹层；同时根据一个 ID，组合出一个 URL，写出到浏览器历史记录里。接下来，如果把弹层关掉，URL 会被重写，也写到浏览器历史记录里。

    重点是，如果在有弹层的显示的同时，刷新页面。这时的行为相当于直接打开根据 ID 组合的那种 URL，这时，我以为会照旧打开页面，不过会根据 URL 的类型来识别出页面状态，发现 URL 里有 ID，就根据 ID 把相应弹层打开。

    但实际上 Pinterest 的做法更巧妙。会直接打开一个样式和弹层差不多的独立页面，相当于为之前弹层里面的内容单独做一个专门的页面。这就免了上面我提到的，根据 URL 的 ID 来识别当前状态，恢复弹层之类的。而且，如果是单独打开弹层出现后的 URL 的用户，肯定都是奔着看大图，而不是瀑布流去的。所以，这样做就很实在很顺了。

  4. 结论可以出来了。
    1. 为了保存当前页面的滚动位置，只能用弹层来显示某张图片的详情。
    2. 弹层出现或者消失，都重写 URL，写入历史记录。
    3. 当直接打开弹层出现时的 URL，就直接进入了真正的某张图片的详情页面。
